## Fixed this pass

`status.success` and `status.warning` in the **light** theme were failing WCAG 1.4.3 when used as text color (e.g. an inline "Saved" or "Warning" message) — `green-600`/`amber-600` only hit 3.15:1 / 3.05:1 against `bg.default` (`gray-50`). Bumped both to the 700 weight (`green-700` `#15803d`, and a newly-added `amber-700` `#b45309`, since amber previously stopped at 600) — both now clear 4.5:1 (4.80:1 / 4.81:1). Dark mode was never affected (these tokens render on a near-black background there, where 400-weight already clears the bar comfortably). `status.danger` was already fine (`red-600`, 4.62:1) and untouched.

This changes the rendered color of anything currently reading `--color-status-success`/`--color-status-warning` in light mode — regenerate `theme.css` (`pnpm --filter @nebula/react-ui generate-tokens`) and eyeball any UI using those tokens before shipping.

## Known failure — not fixed, left for a design decision

`border.default` (`gray-200` on `gray-50` in light, `gray-800` on `gray-950` in dark) fails the 3:1 non-text-contrast bar in both themes. **Left as-is** because:

- No shipped component currently renders it as a *required* control boundary — `primitives`' `Input`/`Textarea`/`Container` are deliberately unstyled (no border color applied), so this token isn't yet load-bearing for identifying any interactive element's edge. WCAG 1.4.11 applies to visual information required to identify a UI component/state, not to purely decorative dividers.
- It is very likely intended to be a subtle decorative divider (card edges, table rules) by design, and bumping it to 3:1 (e.g. `gray-300`/`gray-700`) is a visible aesthetic change, not just a token tweak — that's a design call for whoever builds the styled `Container`/`Input` in `ui`, not something to silently change here.

**Action item for whoever builds those styled components:** if `border.default` ends up as the visible boundary of an `Input`, `Textarea`, `Select`, or similar (i.e. the only thing indicating "this is a text field"), it needs to move to something ≥3:1, or the component needs a second cue (background fill, always-visible label, etc.) that doesn't depend on the border being perceivable.
